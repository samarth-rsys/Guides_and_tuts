# Juniper Startup Port Alarm Robot Test Changes

This note explains the changes made for the Juniper startup port alarm system
test and the reason behind each change.

The test validates this flow:

```text
Robot test
  -> pao-trigger-event generates Juniper PAO "Config is Ready!" syslog
  -> raw syslog is published to Kafka topic fabric-events
  -> Juniper alarm actor decodes the syslog as PAO_CONFIG_READY
  -> actor runs startup port alarm handler
  -> actor asks the switch simulator for physical interfaces
  -> actor maps the CLI port to inventory port data
  -> actor publishes PortStateChange alarm to raw.alarms
  -> events storage actor stores it in persistent_events
  -> Robot verifies the alarm in PostgreSQL
```

## Main Reason For The Changes

RTBrick alarm tests use GELF notification events. Juniper startup port alarms do
not use GELF. They start from a Juniper syslog message.

For this specific alarm, the real device would normally send a PAO message like:

```text
Config is Ready!
```

In the system test we do not use a real Juniper device, so the test must create
that syslog message itself and publish it to Kafka.

## Where The Config Ready Syslog Came From

The `Config is Ready!` message was not guessed from the Robot test. It came from
the Juniper adaptor source code.

The clearest reference is in:

```text
juniper-adaptor/pkg/platform/decoder/syslog_test.go
```

That file has a sample PAO config-ready syslog:

```text
<14>1 2026-01-08T10:42:36.923+01:00 leaf-switch21-91-080-1503-7kec - 16156 - - 2026-01-08 10:42:36,923 leaf-switch21-91-080-1503-7kec INFO: PAORestAgent.py MainThread: Config is Ready!
```

The same test verifies that the decoder reads this message and marks it as:

```text
PAO_CONFIG_READY
```

The production decoder has the same rule in:

```text
juniper-adaptor/pkg/platform/decoder/syslog.go
```

The important logic is:

```text
if message contains "Config is Ready!" and "PAORestAgent"
    DerivedType = PAO_CONFIG_READY
```

So the reason we generated a message containing both strings is simple:

```text
The Juniper adaptor itself says this is the pattern for PAO config-ready.
```

## Does The Device Push Directly To Kafka?

In a real system, the Juniper device does not literally run a Kafka producer.

The easier mental model is:

```text
Juniper device
  -> sends syslog
  -> platform/syslog ingestion puts that syslog on Kafka topic fabric-events
  -> Juniper alarm actor consumes fabric-events
```

In the system test, we skip the real device and the real syslog ingestion layer.
Instead, we directly publish the same syslog bytes into Kafka topic:

```text
fabric-events
```

That direct publish is the test replacement for:

```text
real device syslog -> syslog collector -> Kafka
```

This is why the Robot test says the event is generated by our function and not
by the real device.

## How We Know The Kafka Topic Is fabric-events

The alarm actor startup code is in:

```text
juniper-adaptor/cmd/actors/alarms/main.go
```

There the sidecar is started with:

```text
Topics: ["fabric-events"]
```

That means the Juniper alarm actor listens for incoming alarm-related syslog
messages on `fabric-events`.

The sidecar code is in:

```text
juniper-adaptor/pkg/platform/kafka/sidecar.go
```

It says that for messages from `fabric-events`, or messages that look like
RFC5424 syslog, it attempts syslog decoding:

```text
shouldAttemptSyslogDecode := topic == "fabric-events" OR payload starts with "<"
```

After decoding, it routes by:

```text
MsgID
or
DerivedType
```

For the PAO config-ready message, there is no useful real MsgID, so the decoder
uses `DerivedType = PAO_CONFIG_READY`.

## How We Know PAO_CONFIG_READY Starts Startup Port Alarm

The route is registered in:

```text
juniper-adaptor/pkg/fixtures/alarms/pipeline/pipeline.go
```

The important mapping is:

```text
decoder.PAOConfigReady -> HandleStartupPortAlarm
```

The handler itself is in:

```text
juniper-adaptor/pkg/fixtures/alarms/pipeline/startup_port_alarm.go
```

Its comment and code show that when PAO config-ready is received, it asks the
switch for physical interfaces and publishes startup port state alarms.

That is why the Robot test does not send a normal link-down trap. The startup
port alarm is triggered by PAO config-ready.

## Why The Generated Syslog Looks Like This

The generated test syslog is:

```text
<14>1 <timestamp> <hostname> PAORestAgent.py 16156 - - <timestamp> <hostname> INFO: PAORestAgent.py MainThread: Config is Ready!
```

This shape follows RFC5424 syslog. The beginning is the syslog envelope:

```text
<14>1 <timestamp> <hostname> <app-name> <proc-id> <msg-id> <structured-data>
```

For our test message:

```text
<14>1              -> syslog priority and version
<timestamp>        -> syslog timestamp
<hostname>         -> device hostname used by inventory lookup
PAORestAgent.py    -> application name
16156              -> process id
-                  -> empty MsgID
-                  -> empty structured data
```

Then comes the free-text message body:

```text
<timestamp> <hostname> INFO: PAORestAgent.py MainThread: Config is Ready!
```

The exact body was copied from the adaptor's own decoder test pattern, with the
hostname and timestamp made dynamic for the system test.

The most important parts are:

```text
PAORestAgent.py
Config is Ready!
```

Those are the two strings the decoder uses to classify the message as
`PAO_CONFIG_READY`.

## How We Know It Is Pushed Into Kafka In The Test

The system-test publisher is in:

```text
pkg/lib/pao-trigger-event.go
```

For event names like:

```text
juniper_startup_port_alarm
juniper_startup_port_alarm_*
```

the code calls:

```text
PublishRawKafkaMessage("fabric-events", message, serviceName, timeout)
```

`PublishRawKafkaMessage` creates a Sarama Kafka producer and sends the syslog
bytes as the Kafka message value.

During the Robot run, the helper printed output like:

```text
Generated Juniper startup syslog for <hostname>: <14>1 ...
Published raw Kafka message to fabric-events partition=0 offset=<n>
```

That confirmed the test helper really published the syslog to Kafka.

The Juniper alarm actor logs also showed that it received the same message from
`fabric-events`. That confirmed the message was not only generated locally but
actually consumed by the actor.

## Test Case Added

File:

```text
pkg/juniper/JuniperAlarmTests.robot
```

Added test:

```text
JUNIPER_ALARM_TEST_002_STARTUP_PORT_ALARM_RAISED
```

What the test does:

1. Prepares alarm test tables.
2. Loads Juniper port mapping and inventory data for a BLR3 device.
3. Sends a generated Juniper startup syslog event.
4. Checks PostgreSQL table `persistent_events` for a `PortStateChange_1_3`
   alarm.

Reason:

The test must prove the full system path works, not only that the helper can
send a Kafka message. The final check is done in PostgreSQL because that proves
the alarm actor published the event and the storage actor persisted it.

## BLR3 Device Alignment

The Juniper devices for this test are BLR3 devices.

That matters because these values must all match each other:

```text
Robot inventory setup device
Generated syslog hostname
Device ID
NE / NEP inventory data
Simulator CLI port
Expected alarm description
```

If one of these points to BLR1 and another points to BLR3, the alarm actor may
receive the syslog but fail to find the network element in inventory.

Current test flow uses a BLR3 device name in the Robot event sender:

```text
juniper_startup_port_alarm_BLR3SPINE02
```

The event generator has BLR3 Juniper device config in:

```text
pkg/lib/pao-trigger-event.go
```

Reason:

The Juniper alarm actor looks up the network element by the hostname from the
syslog. Device identity must therefore be consistent from event generation to
inventory lookup.

## Alarm Commands Changes

File:

```text
pkg/lib/AlarmCommands.robot
```

Important changes:

```text
Setup_Port_Mapping_Data
Send_Juniper_Startup_Port_Alarm_Event
Verify_Juniper_Startup_Port_Alarm_In_PostgreSQL
```

`Setup_Port_Mapping_Data` is used to load:

- port mapping catalogue data
- network element data
- network element port data
- network element link data

`Send_Juniper_Startup_Port_Alarm_Event` calls:

```text
send_any_event kafka/a4-kafka:9092 juniper_startup_port_alarm_BLR3SPINE02 ... juniper_actor_test
```

Reason:

The temporary Docker container that runs `pao-trigger-event` must join the
Juniper compose network. If it runs on the RTBrick network, it cannot resolve
or reach the Juniper Kafka container correctly.

The verification checks for:

```text
notificationType LIKE '%PortStateChange%'
perceivedSeverity = 'INFO'
description LIKE '%port 100G_005 is up%'
```

Reason:

The simulator returns the port as operationally up. Startup port alarm logic
publishes:

- `INFO` when the link is up
- `CRITICAL` when the link is down

So the expected alarm for the current simulator data is an `INFO` alarm.

## Events Client Change

File:

```text
pkg/lib/EventsClient.py
```

Changed `send_any_event` to accept an optional Docker network:

```python
docker_network: str = "rtbrick_actor_test"
```

Juniper passes:

```text
juniper_actor_test
```

Reason:

The same Python keyword is shared by RTBrick and Juniper tests. RTBrick should
keep using the default network, while Juniper needs to run the helper container
inside the Juniper Docker Compose network.

## Event Generator Changes

File:

```text
pkg/lib/pao-trigger-event.go
```

Added support for:

```text
juniper_startup_port_alarm
juniper_startup_port_alarm_*
```

The generator creates a raw RFC5424-style syslog message:

```text
<14>1 <timestamp> <hostname> PAORestAgent.py 16156 - - <timestamp> <hostname> INFO: PAORestAgent.py MainThread: Config is Ready!
```

It then publishes the bytes directly to:

```text
fabric-events
```

Reason:

The Juniper alarm actor expects syslog bytes for this path. A GELF event would
not exercise the Juniper startup alarm flow.

The generator uses Sarama directly for this event type.

Reason:

The existing helper path publishes structured PAO/GELF-style events. For this
test, the Kafka value must be the syslog text itself so the Juniper actor can
decode it as syslog and derive `PAO_CONFIG_READY`.

## Switch Simulator Changes

File:

```text
pkg/juniper/pkg/simulator/simulator_handlers.go
```

Added handling for:

```text
show interfaces terse
```

The simulator returns:

```text
et-0/0/68
et-0/0/68.0
lo0
```

Reason:

After PAO config-ready, the startup port alarm handler asks the switch for all
physical interfaces. It keeps real physical Juniper ports and skips things like
subinterfaces and loopback interfaces.

This means:

```text
et-0/0/68    -> used
et-0/0/68.0  -> skipped because it is a subinterface
lo0          -> skipped because it is not an et-* port
```

The `et-0/0/68` port matches the Juniper port mapping catalogue row for:

```text
100G_005
```

That is why the test expects:

```text
port 100G_005 is up
```

## Simulator Test Added

File:

```text
pkg/juniper/pkg/simulator/simulator_test.go
```

Added a test for the new `show interfaces terse` handler.

Reason:

The startup alarm test depends on this simulator command. Without it, the actor
can receive the syslog but fail later because it cannot discover the physical
ports.

The new test uses:

```text
0.0.0.0:0
```

Reason:

Using a fixed port like `8080` can fail when the Docker test stack is already
running and that port is occupied.

## Docker Compose Changes

File:

```text
pkg/juniper/docker-compose.yml
```

Important changes:

```text
ALARM_PRODUCER_TOPIC=raw.alarms
pod-evts-storage-actor
JUNIPER_ALARM_ACTOR_IMAGE
juniper-switch-simulator:testing
```

### Alarm Producer Topic

The Juniper alarm actor is configured with:

```text
ALARM_PRODUCER_TOPIC=raw.alarms
```

Reason:

The Robot test verifies persisted alarm events. The alarm actor must publish to
the topic that the storage actor consumes.

### Storage Actor

Added:

```text
pod-evts-storage-actor
```

It consumes:

```text
raw.alarms
```

Reason:

Without the storage actor, the alarm may be published to Kafka but never appear
in PostgreSQL `persistent_events`. The Robot verification checks PostgreSQL, so
this container is required for the end-to-end test.

### Alarm Actor Image

The compose file uses:

```text
${JUNIPER_ALARM_ACTOR_IMAGE:-juniper-alarm-actor:testing}
```

Reason:

The old published image that was running during debugging received the raw
syslog but treated it as a generic binary message and did not route it to
`HandleStartupPortAlarm`.

The current Juniper adaptor source contains the required startup handler route,
so the local test image must include that newer code. The environment variable
allows CI or a developer to override the image with a proper published tag when
one is available.

### Switch Simulator Image

The compose file uses:

```text
juniper-switch-simulator:testing
```

Reason:

The simulator source was changed to support `show interfaces terse`. The test
must run the rebuilt local simulator image, otherwise the running container may
not include that new handler.

## Important Debug Findings

During debugging, the actor logs showed this behavior with the older alarm
actor image:

```text
Retrieved Message from Kafka topic fabric-events ...
Processing message of type binary
```

But there was no startup alarm handler log.

Meaning:

The syslog was reaching Kafka and the actor container, but the actor image did
not contain the newer routing path needed for startup port alarms.

With the current actor image, the logs reached:

```text
processing startup port alarm for hostname ...
```

That confirmed the test event format was correct and the remaining issues were
inventory/simulator alignment.

## How To Run

From `fabric-system-tests`:

```bash
robot -t JUNIPER_ALARM_TEST_002_STARTUP_PORT_ALARM_RAISED pkg/juniper/JuniperAlarmTests.robot
```

If using local images, make sure these images exist first:

```text
juniper-alarm-actor:testing
juniper-switch-simulator:testing
```

The event helper binary also needs to be rebuilt after changing
`pao-trigger-event.go`:

```bash
go build -o pkg/lib/bin/pao-trigger-event pkg/lib/pao-trigger-event.go
```

## Expected Successful Result

The passing test should find a row similar to:

```text
io.leitstand.actor.events.alarms.PortStateChange_1_3 | INFO | Device <ne-id>: port 100G_005 is up
```

## Common Failure Reasons

### Actor Receives Message But No Alarm Is Created

Likely reason:

```text
The running Juniper alarm actor image does not include the startup alarm route.
```

Fix:

Use a newer published image or build `juniper-alarm-actor:testing` from the
current `juniper-adaptor` source.

### Inventory Lookup Fails

Likely reason:

```text
The syslog hostname and inventory hostname do not match.
```

Fix:

Keep the same BLR3 device identity across Robot setup, `pao-trigger-event.go`,
and the generated syslog hostname.

### Simulator Is Never Called

Likely reason:

```text
The actor stopped before switch access, usually during syslog routing or NE lookup.
```

Fix:

Check actor logs for `processing startup port alarm` and inventory lookup
errors.

### Alarm Published But Not Found In PostgreSQL

Likely reason:

```text
The storage actor is missing or not consuming raw.alarms.
```

Fix:

Make sure `pod-evts-storage-actor` is running and consuming `raw.alarms`.
